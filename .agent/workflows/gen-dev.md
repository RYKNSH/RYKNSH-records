# /gen-dev — ディベート→開発計画の一括生成ワークフロー

// turbo-all

## 概要

ディベート（設計議論）が完了した後に実行する。以下の4文書を階層的に自動生成する：
1. **WHITEPAPER.md** — ディベート成果を統合した経営設計書
2. **ROADMAP.md** — 大きなPhase構成と時系列
3. **MILESTONES.md** — 各Phaseの検証可能な完了条件
4. **TASKS.md** — マイルストーンを実行可能な粒度に分解

## 前提条件

- ディベートが完了していること（ディベートデータ or 設計議論ログが存在すること）
- 対象プロジェクトの `docs/` ディレクトリが存在すること

---

## 1. ディベートデータの収集

対象プロジェクト内のディベート / 設計議論データを収集する：
- `docs/` 配下のディベートログ（例: `holdings_blueprint.md`）
- `strategy_bible.md` 等の既存戦略文書
- 会話コンテキストからの設計方針

## 2. WHITEPAPER.md 生成

ディベートの全合意事項を以下の構造に統合する：

```
# [Project Name] — WHITEPAPER

## Executive Summary（1段落で全体像）
## Philosophy / Vision / Mission（WHY → WHERE → HOW）
## Strategy（WHAT — 組織/収益/顧客獲得/スケール）
## 差別化の核心（プロジェクト固有の技術的・思想的コア）
## Architecture（システム設計 — 図・テックスタック）
## Revenue（プロダクト × 価格 × ターゲット）
## Go-to-Market（チャネル × フェーズ）
## Execution Roadmap（概要レベル）
## Governance（ルール・制約・プロトコル）
```

**ルール**:
- ディベートの結論のみを採用。議論過程は省略
- 全セクションに具体的な数値・KPI・テーブルを含める
- 冗長な説明は排除し、判断に必要な情報だけを残す

## 3. ROADMAP.md 生成

WHITEPAPERのExecution Roadmapを詳細化する：

```
## Phase N: [テーマ名]
- 目的
- 対象コンポーネント
- 完了条件（概要）
- 上位プロジェクトの責務（該当する場合）
```

**ルール**:
- 各Phaseに明確な依存関係を持たせる
- 現在地を明示（✅/🔶/⬜）
- 並列実行可能なPhaseは明示する

## 4. MILESTONES.md 生成

ROADMAPの各Phaseを検証可能なマイルストーンに分解する：

```
### [ID]-[番号]: [マイルストーン名]
- **Done**: 検証可能な完了条件（曖昧な表現禁止）
- **前提**: 依存するマイルストーンID
- **現状**: ✅/🔶/⬜
- **KPI**: 定量目標（あれば）
```

**ルール**:
- 完了条件は「〜が存在する」「〜が動作する」のように客観的に検証可能
- 最後に Mermaid で依存関係図を追加

## 5. TASKS.md 生成

各マイルストーンを **1セッション以内に完了可能な粒度** に分解する：

```
| ID | タスク | 状態 | 担当 | 備考 |
|----|--------|------|------|------|
| [MS-ID]-01 | [具体的作業] | ⬜ | [担当] | [依存関係] |
```

**ルール**:
- 1タスク = 1コミット or 1PRに対応する粒度が理想
- 最後に「Phase別サマリー」と「今すぐ着手可能なタスク」セクションを追加

## 6. 相互リンクの設定

4文書の末尾に他文書へのリンクを追加。

## 7. ユーザーにレビュー依頼

生成した4文書を提示し、レビューを依頼する。
